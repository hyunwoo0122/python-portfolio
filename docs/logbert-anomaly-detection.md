# ログ異常検知・自動分析システム

> 業務上の理由によりソースコードの公開はできませんが、システムの概要・技術選定の理由・成果をまとめています。

---

## 背景・課題

顧客が使用する製品は独自形式のログ（Linuxベース）を出力しており、  
以下のような構造になっていました。

```
[INFO] 2024-01-15 10:23:41 register_name 処理内容メッセージ
```

このログは専門知識がなければ解読が困難なため、  
**問題が発生するたびに顧客が会社にログを送付し、エンジニアが原因を調査する**  
という非効率なフローが続いていました。

### 解決したかった課題
- 顧客が自力でログの異常原因を把握できない
- エンジニアへの問い合わせ対応コストが高い
- ログ解析に専門知識が必要

### 目指したゴール
ログを入力するだけで、**知識がない人でも異常の原因と内容を理解できる**システムの構築

---

## システム概要

LogBERTによる異常検知とRAGベースの原因分析を組み合わせた、  
ログ自動解析システムです。

```
ログ入力
  ↓
LogBERT（異常検知）
  ↓
異常ログの抽出
  ↓
分析に必要なログの別途抽出
  ↓
Ollama（GPT-OSS 120B）へ入力
  ↓
埋め込み処理 → ベクトルDBへ保存
  ↓
RAG（質問時刻のログを検索・取得）
  ↓
原因分析結果の出力
```

---

## 技術スタック

| 技術 | 用途 | 選定理由 |
|------|------|---------|
| Python | 全体実装 | チームの共通言語 |
| LogBERT | ログ異常検知 | Transformerベースで文脈を考慮した検知が可能。シーケンスベースのDeepLogより高精度 |
| LangChain | LLM連携・パイプライン構築 | 各処理をチェーンとして接続でき、実装効率が高い |
| RAG | 関連ログの検索・取得 | 質問時刻に対応するログをベクトル検索で高速に取得できる |
| Ollama | ローカルLLM実行基盤 | GPT-OSS 120Bをローカル環境で動作させ、データを外部送信せずに分析 |

### LogBERTを選んだ理由（DeepLogとの比較）

| | DeepLog | LogBERT |
|--|---------|---------|
| モデル構造 | LSTMベース（シーケンス） | Transformerベース（文脈考慮） |
| 文脈理解 | 限定的 | 前後の文脈を考慮した検知が可能 |
| 精度 | ベースライン | 本プロジェクトでより高精度を達成 |

## 成果

| 指標 | 結果 |
|------|------|
| Precision | **87%** |
| Recall | **90%** |

### 達成までの過程

1. 初期学習結果のPrecision・Recallを計測
2. トレードオフを分析し、改善対象を特定
3. 以下のパラメータを段階的に調整：
   - 異常判定の閾値（threshold）
   - 学習データの前処理（外れ値の除去）
   - 特徴量の次元数
4. 各調整後に再評価し、最終的な数値を達成

### Precision・Recallのバランスについて

異常検知システムでは以下のトレードオフがあります：

- **Precisionが低い** → 正常ログを異常と誤判定（誤検知が多い）
- **Recallが低い** → 実際の異常を見逃す（検知漏れが多い）

本システムは「**見逃しを減らすこと**」を優先し、  
Recallをやや高めに設定しました。  
顧客に対して「異常を見逃さない」ことが信頼性の観点から重要と判断したためです。

## 工夫した点・技術的チャレンジ

### 1. LogBERTのカスタマイズ
GitHubの公開実装をベースに、対象ログの形式に合わせてカスタマイズを実施。  
独自形式のログをモデルが処理できる形に変換する前処理を設計しました。

### 2. プロンプト設計
LLMへの入力プロンプトを構造化し、以下の形式で出力されるよう設計：

```
1. 検出された異常の概要
2. 推定される原因
3. 推奨される対処法
```

論理的な順序で出力されるよう、プロンプトを複数回改善しました。

### 3. ローカル環境での完結
顧客ログはセンシティブなデータを含むため、  
Ollamaを使用してすべての処理をローカル環境内で完結させました。  
外部APIへのデータ送信なしに高精度な分析を実現しています。
